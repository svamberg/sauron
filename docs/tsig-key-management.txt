TSIG Key Management with keygen
================================

SYNOPSIS
--------
  keygen [OPTIONS]
  keygen --setmasterkey[="<passphrase>"]
  keygen --add="<server>,<keyname>,<keytype>,<keylen>[,<comments>,<passphrase>]"
  keygen --del=<server>,<keyname>
  keygen --regen=<server>,<keytype>
  keygen --list[=<server>]

DESCRIPTION
-----------
The keygen utility is used for creating and managing TSIG (Transaction SIGnature)
keys for DNS authentication. TSIG keys are used to secure DNS transactions such
as zone transfers (AXFR/IXFR) and dynamic updates (DDNS).

Key features:
  * Generate cryptographically secure TSIG keys
  * Store keys encrypted in the database
  * Master passphrase protection for all keys
  * Support for automatic and static key modes
  * HMAC-MD5 algorithm support

PREREQUISITES
-------------
Before using keygen, you must:

1. Set up the Sauron master key (required for all operations):
   ./keygen --setmasterkey

   This creates a .key file alongside your configuration file and encrypts
   all stored keys using RC5 encryption with the master passphrase.

2. Ensure the DNSSEC key generation program is available and configured:
   SAURON_DNSSEC_KEYGEN_PROG must point to a valid key generation tool
   (e.g., dnssec-keygen from BIND).

3. Have appropriate database access permissions.

MASTER KEY SETUP
----------------
The master key protects all TSIG keys stored in the database.

Interactive setup:
  ./keygen --setmasterkey

Non-interactive setup:
  ./keygen --setmasterkey="MySecretPassphrase"

When changing the master key on a system with existing keys:
  - The tool will warn you to backup the database first
  - All existing keys will be re-encrypted with the new passphrase
  - The operation requires confirmation (Y to proceed)

Warning: Always backup your database and configuration before changing
the master key!

CREATING A NEW TSIG KEY
-----------------------
Interactive mode:
  ./keygen --add

  You will be prompted for:
  - Server name: The server this key is associated with
  - Key name: Unique identifier for the key (e.g., "transfer-key", "ddns-key")
  - Key type: TSIG (default, only supported type)
  - Key length: 1-512 bits (default: 128)
  - Comments: Optional description

Non-interactive mode:
  ./keygen --add="server,keyname,type,length,comments,base64key"

  Parameters:
  - server      : Server name or ID
  - keyname     : Unique key name (alphanumeric, no spaces)
  - type        : Key type (TSIG)
  - length      : Key length in bits (1-512)
  - comments    : Optional comment/description
  - base64key   : Optional pre-generated key in Base64 format

Examples:
  # Generate a new 256-bit TSIG key for server "ns1"
  ./keygen --add="ns1,transfer-key,TSIG,256,Zone transfer key"

  # Generate with default length (128 bits)
  ./keygen --add="ns2,ddns-key"

  # Use a pre-generated key (Base64 encoded)
  ./keygen --add="ns1,imported-key,TSIG,0,Imported key,AQ==abcdef123456=="

LISTING KEYS
------------
List all keys for a server:
  ./keygen --list=servername

List keys of specific type:
  ./keygen --list="servername,TSIG"

Example output:
  Key name                  Type     Bits Mode   Comments
  ------------------------- -------- ---- ------ ------------------------------
  ddns-key                  HMAC-MD5  128 Auto   Dynamic update key
  transfer-key              HMAC-MD5  256 Static Zone transfer

Mode meanings:
  Auto   - Key was auto-generated by keygen
  Static - Key was manually provided (static/imported)

DELETING A KEY
--------------
Interactive mode:
  ./keygen --del

  You will be prompted for:
  - Server name
  - Key name to delete

Non-interactive mode:
  ./keygen --del=server,keyname

Example:
  ./keygen --del="ns1,old-transfer-key"

Warning: Deleting a key will break any DNS configurations using that key!
Ensure the key is removed from all DNS server configurations first.

REGENERATING KEYS
-----------------
Regenerate all auto-generated keys (mode=Auto) for a server:

Interactive:
  ./keygen --regen

Non-interactive:
  ./keygen --regen=server,type

Example:
  ./keygen --regen="ns1,TSIG"

This regenerates keys while preserving key names and associations.
Useful for periodic key rotation for enhanced security.

KEY TYPES AND ALGORITHMS
------------------------
Supported key types:
  TSIG - Transaction SIGnature for DNS authentication

Supported algorithms:
  HMAC-MD5 (algorithm 157) - Most widely supported for TSIG

Key length recommendations:
  - Minimum: 64 bits (not recommended for production)
  - Standard: 128 bits (default, adequate for most uses)
  - Enhanced: 256 bits (recommended for high-security environments)
  - Maximum: 512 bits

DATABASE STRUCTURE
------------------
Keys are stored in the 'keys' table with the following relevant columns:

  id          - Unique identifier
  type        - Key type (1 = server key)
  ref         - Server ID reference
  name        - Key name (unique per server)
  protocol    - Protocol (2 = TSIG)
  algorithm   - Algorithm ID (157 = HMAC-MD5)
  mode        - 0 = Auto-generated, 1 = Static/Manual
  keysize     - Key length in bits
  secretkey   - Encrypted key data (Base64)
  comment     - User comment/description

The secretkey is encrypted using RC5 with the master passphrase.

USING KEYS WITH DNS SERVERS
---------------------------
After creating a TSIG key in Sauron, you need to configure your DNS server.

For BIND (named.conf):

  key "transfer-key" {
    algorithm hmac-md5;
    secret "base64-encoded-key-from-sauron";
  };

  server 192.168.1.1 {
    keys { "transfer-key"; };
  };

To retrieve the actual key value from the database, use the Sauron web
interface or query the database directly (requires decryption with master key).

EXPORTING KEYS FOR DNS CONFIGURATION
------------------------------------
To use TSIG keys with DNS servers, you need to export them from Sauron.
This can be done via:

1. Sauron web interface - View key details and copy the Base64 secret
2. Direct database query (requires manual decryption)
3. Custom export scripts using Sauron::BackEnd module

Example workflow for zone transfer configuration:
  1. Create key: ./keygen --add="ns1,axfr-key,TSIG,256,AXFR authentication"
  2. Retrieve key secret via web interface
  3. Configure primary DNS server with the key
  4. Configure secondary DNS server with the same key
  5. Test zone transfer with: dig @primary axfr example.com

BEST PRACTICES
--------------
1. Master Key Security:
   - Use a strong, unique passphrase for the master key
   - Store the .key file with restricted permissions (mode 600)
   - Backup the master key separately from database backups

2. Key Rotation:
   - Regenerate TSIG keys periodically (e.g., annually)
   - Use --regen for automated key rotation
   - Document key creation/rotation dates in comments

3. Key Naming:
   - Use descriptive names indicating purpose:
     * axfr-key, ixfr-key (zone transfers)
     * ddns-key, update-key (dynamic updates)
     * <zone>-key (zone-specific keys)

4. Access Control:
   - Limit key regeneration/deletion to authorized administrators
   - Audit key usage through Sauron logs
   - Use different keys for different purposes/servers

5. Backup Strategy:
   - Backup database BEFORE changing master key
   - Export critical keys to secure offline storage
   - Test key restoration procedures

TROUBLESHOOTING
---------------
"Sauron master key not defined"
  Run ./keygen --setmasterkey first.

"Cannot execute: $SAURON_DNSSEC_KEYGEN_PROG"
  Ensure the key generation program is installed and executable.
  Check SAURON_DNSSEC_KEYGEN_PROG configuration option.

"Key with same name already exists"
  Choose a different key name or delete the existing key first.

"invalid/unsupported key type"
  Only TSIG is currently supported.

"failed to generate key"
  Check that the external key generation program works correctly.
  Verify /tmp has sufficient space for temporary files.

"existing keyfile with unsafe permissions"
  The .key file must have mode 600 (owner read/write only).
  Fix with: chmod 600 /path/to/config.key

FILES
-----
  /opt/sauron/keygen              Main script
  /opt/sauron/config.key          Encrypted master key (created by --setmasterkey)
  /tmp/sauron.*                   Temporary directory for key generation

EXAMPLES
--------
# Initial setup - create master key
./keygen --setmasterkey="SecurePassphrase123"

# Create a zone transfer key
./keygen --add="ns1,zone-transfer,TSIG,256,Primary-Secondary AXFR"

# Create a dynamic update key
./keygen --add="ns1,ddns-update,TSIG,128,DHCP server DDNS"

# List all keys for a server
./keygen --list=ns1

# Regenerate all auto-generated keys
./keygen --regen="ns1,TSIG"

# Delete an obsolete key
./keygen --del="ns1,old-key"

# Complete workflow for new DNS server
./keygen --setmasterkey="MasterSecret"
./keygen --add="dns1,axfr-key,TSIG,256,Zone transfer authentication"
./keygen --add="dns1,ddns-key,TSIG,128,Dynamic updates from DHCP"
./keygen --list=dns1
